# NestSuite 設計者向けガイドライン

> 作成: v2.17.1
> 目的: 設計者・課題整理担当者・実装プロンプト作成者・レビュー方針作成者・引継ぎ担当者が共通して守る恒久規約を一か所に整理する。開発者向けガイドライン（`docs/development/nestsuite-development-guidelines.md`）と対になる、設計側の正本文書。

---

## 1. 文書の目的

この文書は、NestSuite の以下の役割に適用されます。

- 設計者
- 課題整理担当者（backlog 整理）
- 実装プロンプト作成者
- レビュー方針作成者
- 引継ぎ担当者（設計者交代時の引継ぎ側・受け取り側の両方）

**主な目的:**

- 設計判断のぶれを防ぐ
- 開発者向け規約の参照漏れを防ぐ
- 引継ぎプロンプトを必要以上に肥大化させない
- 生成AIのトークン消費と誤実装を抑える
- 恒久規約と一時的な指示を分離する
- 完了・保留・未着手の状態を混同しない

**今回の指示とこの文書が矛盾する場合は、今回の指示を優先してください。** ただし、その場合は「何を」「なぜ」「今回に限って」上書きするのかを明示してください（§2 参照）。

---

## 2. 正本文書の優先関係

NestSuite の設計・実装に関わる文書は、次のように役割が分かれます。

```text
設計者向け恒久規約:
docs/development/nestsuite-designer-guidelines.md

開発者向け恒久規約:
docs/development/nestsuite-development-guidelines.md

現在地・一時的判断:
引継ぎプロンプト

今回固有の実装差分:
個別実装プロンプト
```

優先関係は、原則として次の順とします（上が優先）。

```text
1. 個別プロンプトの明示的な一時上書き
2. 引継ぎプロンプトの現在地・確定判断
3. 設計者向けガイドライン（この文書）
4. 開発者向けガイドライン
5. その他の参考文書（docs/design, docs/planning 等）
```

**暗黙の上書きは禁止します。** 個別プロンプトが規約と異なる指示を出す場合は、次を明示してください。

- 何を上書きするか（対象ルール・対象文書）
- なぜ上書きするか（理由）
- 今回に限ってか、恒久的な規約変更かを明示する（恒久変更の場合は該当ガイドラインの改訂として扱う）

---

## 3. 開発者向けガイドラインの遵守

- 設計者は、すべての実装プロンプトの冒頭で、実装者に対し次を確認・遵守させることを明記してください。

```text
docs/development/nestsuite-development-guidelines.md
```

- 設計者自身も、実装指示を作成する前に開発者向けガイドラインを確認してください。
- 開発者向けガイドラインに既に存在する共通事項（外部依存禁止、1バージョン1目的、テスト整合性原則 等）は、個別プロンプトへ**原則として全文を再掲しません**。参照リンクで足ります。
- 開発者向けガイドラインと矛盾する指示を個別プロンプトに書く場合は、§2 の上書きルールに従ってください。

---

## 4. 個別実装プロンプトの原則

個別実装プロンプトには、**原則として次だけ**を書きます。

- version
- 今回の結論
- 目的
- 対象範囲
- 今回固有の仕様
- やること
- やらないこと
- 規約にないテスト観点
- 必要な docs・release notes・backlog 更新
- 一時的な規約上書き（ある場合のみ、§2 の形式で）
- 完了条件
- 作業対象者が通常エンジニアかエキスパートか（§7 参照）

次のような共通事項は、開発者向けガイドラインに存在する限り、**毎回全文を繰り返しません**。

- 外部依存を追加しない
- 既存テストを削除・skip しない
- 1バージョン1目的
- version 更新箇所
- schema 変更時の一般ルール
- release notes の一般ルール
- テストクラス命名規則
- 静的テスト追加基準
- 実装範囲を勝手に広げない

ただし、今回の事故防止に特に重要な制約は、短く再掲して構いません（例:「今回は schema を変更しない」など、事故が起きやすい 1〜2 行に限る）。

---

## 5. 生成AI向けプロンプト設計

生成AIによる実装を前提として、次を守ってください。

- 長い背景説明より、対象・差分・禁止事項を優先する
- 同じ制約を複数箇所で繰り返さない
- 対象ファイルを事前に断定しすぎず、まず既存実装を確認させる
- 調査だけで終わらせず、実装課題では実装まで完了させる
- 不明点を理由に無関係な設計整理へ広げない
- 既存責務を確認してから、新しいサービスやクラスを追加する
- 共通化は、重複が存在するだけで自動採用しない（統一しない判断も有効な設計判断として扱う）
- 将来拡張のためだけの抽象化を追加しない
- 作業報告では一般規約を再掲せず、変更内容と結果に集中する
- トークン節約のため、全ファイルの説明や不変事項の大量再掲を要求しない

---

## 6. 課題選定と version 管理

- 1バージョン1目的を基本とする
- 通常 backlog、トリガー待ち、完了済みを区別する
- 未完了であることだけを理由に、トリガー待ち課題へ着手しない（着手トリガーが成立しているかを個別に確認する）
- 完了済み課題を backlog へ戻さない
- 完了履歴は release notes で管理する
- 完了済み番号を再利用しない
- docs 整理を機能実装へ便乗させない（別 version として分離する）
- 緊急修正、リリース前補正、通常 version を区別する
- version を決める前に、現在の正式到達点（最新 release notes・backlog の状態）を確認する
- 複数候補がある場合は、利用者・開発者の認知負荷、リスク、保存形式への影響、実装規模を比較してから選定する

---

## 7. 通常エンジニアとエキスパートの使い分け

**通常エンジニア向け:**

- 小規模 UI 改善
- 既存設計内の実装
- 責務が明確なリファクタリング
- docs・テスト整理
- 明確な回帰修正

**エキスパート向け:**

- 保存形式変更
- session・draft・復元などデータ保護設計
- 複数責務をまたぐ構造変更
- 前方互換・後方互換設計
- 大規模な責務再配置
- 実装前に複数案比較が必要な課題
- 既存設計の安全性評価

エキスパートレビューを実施した場合でも、**特別進行を恒常化しません。** エキスパート対応フェーズが完了したら、以降は通常 backlog 運用に戻します（v2.17.0 での運用復帰を参照）。

---

## 8. 引継ぎルール

設計者の引継ぎでは、引継ぎプロンプトとは別に、必ず次の 2 文書を参照させてください。

```text
docs/development/nestsuite-designer-guidelines.md
docs/development/nestsuite-development-guidelines.md
```

引継ぎプロンプトには恒久規約を全文複製せず、次を中心に記載してください。

- 正式リポジトリ
- 最新 version
- 未リリースかリリース済みか
- 現在の第一候補
- 直近で完了した事項
- 再オープン禁止事項
- トリガー待ち事項
- 現在の不変条件
- 一時的な例外
- 次に必要な具体作業

**引継ぎプロンプトだけで開発規約を置き換えません。** 引継ぎプロンプトは「現在地」を伝えるものであり、恒久規約の正本は常に設計者向け・開発者向けの両ガイドラインです。

---

## 9. 設計判断と docs

- 恒久的なルールは `docs/development/` 配下のガイドラインへ置く
- 個別機能の設計判断は `docs/design/` または `docs/planning/` 文書へ置く
- 一時的な作業指示はプロンプトへ置く
- 完了履歴は `docs/release-notes.md` へ置く
- 未着手・保留・将来候補は `docs/backlog.md` へ置く
- 同じ事実を複数文書へ重複して正本化しない（参照リンクを使う）
- 文書を追加する前に既存の正本がないか確認する
- stale docs を発見しても、今回の目的と無関係なら作業範囲を広げず、必要に応じて別課題化する

---

## 10. レビュー時の確認

設計者は PR や実装結果を確認するとき、少なくとも次を確認してください。

- 個別プロンプトの目的を達成しているか
- 開発者向けガイドラインに違反していないか
- 作業範囲が広がっていないか
- 保存形式や version が意図せず変わっていないか
- テストを削除・skip していないか
- release notes と backlog の状態が正しいか
- 完了済み課題を再登録していないか
- 実行できなかった確認を、成功済みとして扱っていないか
- GitHub connector 等で結果を確認できない場合、「未実行」と「確認不能」を区別しているか
- PR タイトル・本文が実際の変更内容と一致しているか

---

## 11. 設計者向け完了報告

設計者が実装プロンプトを作る場合やレビューする場合、報告は次を中心に簡潔にまとめてください。

- 結論
- 採用案と理由
- 対象 version
- 対象範囲
- 重要な非対象
- テスト結果
- release 可否
- 次の候補

一般規約の全文再掲は不要です。

---

## 12. 開発者向けガイドラインとの役割分担

| 観点 | 設計者向け（この文書） | 開発者向け |
|------|----------------------|-----------|
| 対象読者 | 設計者・課題整理担当者・プロンプト作成者・引継ぎ担当者 | 実装者（プロンプト受け手） |
| 主な内容 | 課題選定・version管理・引継ぎ・レビュー方針・プロンプト設計 | 実装ルール・保存形式・UI規約・テスト規約・報告形式 |
| 参照関係 | 実装ルールの詳細は開発者向けガイドラインを参照する | 課題選定・引継ぎの詳細は設計者向けガイドラインを参照する |

両文書は互いの正本を重複させず、参照リンクで連携します。
