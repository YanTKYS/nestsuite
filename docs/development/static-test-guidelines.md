# 静的テスト持続可能性ガイドライン（TD-73）

> 作成: v2.16.23 TD-73
> 目的: docs-contract test（`docs/*.md` の内容を検証するテスト）と、ソーステキスト静的確認テスト（`.cs` ファイルを文字列として読み込み、構造や配線を検証するテスト）の書き方の基準を示し、docs やコードが自然に成長するだけで CI が壊れることを防ぐ。

この文書は「静的テストを禁止する」ための文書ではありません。WPF の `Window` を直接インスタンス化できない、`private` 実装の配線を確認したい、といった場面では静的テストが最も現実的な手段になることがあります。
一方で、review3-fable5.md / review4-fable5.md で指摘されたとおり、v2.16.14〜v2.16.19 の間に静的テストが実際に 3 回 CI を壊しました。原因はいずれもテストの書き方にあり、テストが検証しようとしていた設計そのものは正しいままでした。この文書は、その再発を防ぐための知見をまとめたものです。

---

## 1. 静的テストの位置づけ

- NestSuite では、WPF / Shell partial / docs-contract の性質上、ソーステキストや docs テキストを確認するテストを使う場合がある
- 静的テストは有用だが、実装詳細や自然な文書成長に強く結びつくと壊れやすい
- 原則は、挙動テスト・純粋 helper の単体テスト・公開 API に近いテストを優先する
- 静的テストは、直接挙動確認が難しい境界を守るための補助として使う

判断の優先順位は次のとおりです。

1. 実際に呼び出して結果を検証できる挙動テスト
2. UI 非依存の `static` helper に切り出した上での単体テスト
3. `public` / `internal` API に近い形でのリフレクション確認
4. どうしても難しい場合のみ、ソーステキスト静的確認

---

## 2. docs-contract test の推奨パターン

- 重要語句の存在確認に留める（`Assert.Contains("キーワード", text)` 程度）
- 文言完全一致ではなく、概念が記録されていることを確認する
- release notes は見出しアンカーで対象セクションを抽出する
- `"— TD-xx:"` や `"— SH-xx:"` のような見出し専用パターンを使う（本文中の ID 言及と衝突しない）
- backlog の完了済み範囲は helper で範囲解釈する（`"TD-A〜TD-B"` を正規表現でパースし、対象 ID が範囲内かを判定する）
- CRLF / LF の違いに依存しない
- 見出しや明確なセクション境界を使う
- 変更されやすい説明文ではなく、安定した ID・見出し・重要語句を確認する

## 3. docs-contract test の避けるべきパターン

- release notes 全体に対して `IndexOf("TD-66")` のように bare ID だけで探す
- 後続セクション本文中の過去 ID 参照に誤ヒットする検索
- `"\n"` 固定など改行コードに依存するテスト
- backlog の `"TD-64〜TD-70"` のような範囲表記を文字列完全一致で確認する
- 説明文の一字一句を固定するテスト
- docs の自然な追記・整理・順序変更で壊れるテスト
- 「ある文言が存在する」だけでなく「ある文言が存在しない」ことを広範囲に確認しすぎるテスト

---

## 4. ソーステキスト静的確認の推奨パターン

- 挙動テストが難しい理由をコメントで明記する
- method signature や明確な境界（次のメソッド宣言など）で対象範囲を切り出す
- private 実装全体ではなく、守りたい境界だけを確認する
  - 例: `SaveSession` を復元処理中に直接呼ばない
  - 例: `WorkspaceKind` を復元判定の信頼ソースにしない
  - 例: `ShowError` と `Confirm` を同じ分岐で二重表示しない
- 文言リテラルは可能なら builder / const / helper 側で挙動テストする
- 静的確認は「設計境界の固定」に使い、細かい実装順序の固定に使わない

## 5. ソーステキスト静的確認の避けるべきパターン

- private method の細かい実装順序を過度に固定する
- リファクタリングだけで壊れるような検索
- UI 文言リテラルを Shell partial 内で直接探す（builder / const に切り出せないか先に検討する）
- helper に切り出せる内容を切り出さずにソーステキスト確認する
- 1 つの巨大な partial / test class に多数の静的確認を積み増す
- コメントや説明文だけにヒットしても通る曖昧なテスト

---

## 6. 静的テストを追加する前の確認

1. 挙動テストで確認できないか
2. UI 非依存 helper に切り出して単体テストできないか
3. public / internal API に近い形で確認できないか
4. そのテストが守る設計境界は何か
5. docs やコードの自然な整理で壊れないか
6. 重要語句確認で足りるか、文言完全一致が本当に必要か
7. 失敗したとき、開発者が何を直せばよいか分かるメッセージになっているか

---

## 7. 既存の良い例・注意例

過去の失敗を責めるためではなく、同じ形の判断を今後も再利用できるようにするための実例です。

**良い例**

- `ExtractReleaseNotesSection`（`NestSuiteDocsContractTests.cs`）: `"— TD-xx:"` の見出しアンカー方式で対象セクションを抽出する。本文中に過去 ID への言及が増えても壊れない
- `BacklogCompletedRangeCoversTD`（同上）: backlog の完了済み範囲行を正規表現でパースし、対象 ID が範囲に含まれるかを判定する。範囲表記が今後どう圧縮されても壊れない
- `SessionRestoreFailuresMessageBuilder`（`NestSuite/Services/`）: 通知文言の組み立てを UI 非依存の `static` helper に切り出し、`MultipleOpenFailureMessageBuilderTests.cs` 等で実際の入出力を挙動テストする。ソーステキスト確認は「Shell 側がこの helper を呼んでいるか」という配線確認だけに絞れる
- `NotifyRestoreFailures` の静的確認（`SessionTabMapperTests.cs`）: `ShowError` / `Confirm` の呼び出し回数・分岐構造だけを確認し、文言そのものは builder のテストに任せる

**注意例（過去に CI を壊した実例）**

- bare `"TD-66"` を `IndexOf` して release notes セクションを抽出しようとしたところ、後続バージョンの本文が過去 ID を言及したことで誤検出した（→ 見出しアンカー方式へ移行）
- `"\n    }\n"` のような改行コード固定の境界検出が、Windows ランナーの CRLF チェックアウトで一致しなくなった（→ 次のメンバー宣言など、改行コードに依存しない境界へ変更）
- backlog の完了済み範囲表記（`"TD-64〜TD-67"` 等）が新しい ID の追加で圧縮され、期待していたリテラルが消えた（→ 範囲解釈 helper へ移行）
