# Backlog — NestSuite

## 1. この文書の役割

`docs/backlog.md` は以下のみを管理する。

- **未着手・保留・将来候補**の実装候補
- **長期構想・保留**（`LT-` prefix）
- **見送り・採用しない方針**（`RJ-` prefix）

この文書が管理しないもの:

- **完了済み項目** → `docs/release-notes.md` で管理する
- 取り消し線で残した完了項目（積み上げない）
- `<details>完了済み</details>` セクション（作成しない）

完了した項番は **欠番** として扱い、再利用しない。  
実装履歴を確認する場合は `docs/release-notes.md` を検索する。

v2.18.23（TD-89）で全未完了項目の採用・見送り・着手トリガーを総点検した。各項目の分類理由・削除履歴は `docs/planning/backlog-adoption-trigger-review.md` を参照。

## 2. 採番・追記ルール

### prefix 一覧

| prefix | 対象セクション | 例 |
|--------|----------------|----|
| `SH-` | NestSuite Shell 改善 | `SH-1` |
| `TN-` | TempNest 改善 | `TN-3` |
| `L` / `M` / `H` | NoteNest Workspace 改善（難易度別） | `L4` / `M2` / `H3` |
| `ID-` | IdeaNest Workspace 改善 | `ID-4` |

| `TD-` | 技術的負債・保守性 | `TD-33` |
| `LT-` | 長期構想・保留 | `LT-2` |
| `RJ-` | 見送り・採用しない方針 | `RJ-1` |

### 必須ルール

- 新規項目は該当 prefix のセクション末尾に追加する
- 完了した項目は backlog から削除し、`docs/release-notes.md` に記録する
- 完了済み項番は再利用しない（欠番として扱う）
- 取り消し線で完了項目を残さない
- `<details>完了済み</details>` を追加しない
- 複数体系にまたがる項目は主責務のセクションへ置き、関連 ID を概要に記載する
- 長期構想・保留は `LT-` 項番で管理する
- 見送り・採用しない方針は `RJ-` 項番で管理する
- `LT-` / `RJ-` も採番済み番号は再利用しない
- 完了時は `docs/release-notes.md` に backlog ID と実装内容を記録する

### 運用ルール（v2.18.23 TD-89 で追加）

- **着手トリガーのない長期候補を追加しない**。トリガーは「実利用で同一不満が報告された」「実測で基準を超過した」など観測可能な条件で書く（「必要になったら検討」は不可）
- **完了済み・見送り済み ID を再利用しない**。見送り項目を再検討する場合は新 ID を採番する
- **実装主体を明記する**（通常エンジニアで実装可能／着手時に設計レビュー必要／エキスパート設計後に実装）
- **設計レビュー済みかどうかを概要に明記する**（レビュー文書があれば参照先を書く）
- **実利用評価が必要な項目を推測で着手しない**（「実機評価後に判断」「実利用の声を待つ」項目は声が観測されるまで実装対象へ繰り込まない）
- 保存形式変更を伴う項目は、着手トリガー成立に加えて FM-1（`docs/architecture/schema-versioning-policy.md`）準拠の最小設計成立を着手条件とする

### 優先度の定義

- **A**：現行運用上の体感改善が大きく、既存設計への影響が小さい
- **B**：有益だが、設計整理や複数 Workspace への影響確認が必要
- **C**：長期的に有望だが、現時点では慎重に扱う

NoteNest Workspace の改善では「WPF 標準 TextBox の範囲内かどうか」を追加判断基準として参照すること。
