# NestSuite 魅力向上の方向性

## この文書の位置づけ

この文書は、確定仕様や実装計画ではない。次の位置づけとする。

- NestSuite の中期的な価値向上を検討するための設計方針
- backlog 項目を選定するときの判断材料
- 認知負荷軽減後の次のテーマ候補
- 個別機能の実装を直接指示する文書ではない
- 保存形式や Workspace 構造の変更を決定する文書ではない

実装が必要になった場合は、通常の backlog 項目として個別に起票・検討する。この文書自体は特定バージョンでの一括実装を指示しない。

---

## 1. 背景

NestSuite では、これまで次を主な改善テーマとしてきた。

- 利用者の認知負荷軽減
- 開発者の認知負荷軽減
- UI や導線の簡素化
- 保存・復元・データ保護の強化
- 開発文書や責務境界の整理
- 不要な機能・表示・旧資産の縮退

これらにより、次の基盤はおおむね整った。

- 迷いにくい
- 壊れにくい
- 保存への不安が少ない
- 複数の思考形式を 1 つのアプリで扱える
- 開発時の判断根拠が整理されている

今後も認知負荷軽減を基礎方針として維持する。一方で、次の段階では、負担を減らすだけでなく、利用者が NestSuite を再び開きたくなる価値を検討する。

---

## 2. 魅力向上の定義

NestSuite における魅力向上を、派手な UI や機能数の増加とは定義しない。次の状態を魅力向上とする。

- 書いた内容が後から再び役に立つ
- 断片的な思考を次の形へ進められる
- 過去の内容を自然に再発見できる
- NestSuite を開くと次に行うことが見つかる
- NestSuite 内で考えたものを成果物として外へ出せる
- 複数の Nest が単なる同居ではなく、明示操作によって緩やかにつながる
- 軽さ、閉域適性、単一 EXE 志向を維持したまま価値が高まる

中心となる方向性を、次のように記載する。

> 蓄積した思考を、再発見し、次の形へ進められるツール

---

## 3. 評価軸の変化

これまでの主な評価軸:

- 迷わないか
- 壊れないか
- 分かりにくくないか
- 保存や復元に不安がないか
- 開発者が責務を追いやすいか

今後追加する評価軸:

- また開きたくなるか
- 過去の内容が再び役に立つか
- 次の行動につながるか
- NestSuite だから成立する体験か
- 機能追加によって認知負荷を再び増やしていないか
- 明示操作で制御できるか
- 外部依存なしで実現できるか

従来の評価軸を捨てるのではなく、その上に追加する。

---

## 4. 有望な方向性

### 4.1 書いたものが再び役に立つ

蓄積した情報を自然に再発見できる体験を検討する。候補:

- 最近よく触ったテーマ
- 今日・今週更新したファイル
- しばらく開いていないノート
- 未整理の IdeaNest カード
- `[TODO]` や `[NOTE]` が残る NoteNest
- 同じキーワードを含む別 Workspace
- 最近使用した Workspace
- TempNest に残っている断片

この方向性の価値:

> NestSuite を開くと、次に何を考えればよいか分かる

自動分析や高度な推薦を前提とせず、既存情報から取得できる内容を優先する。

### 4.2 思考が Workspace 間を流れる

各 Workspace の疎結合は維持する。ただし、利用者の明示操作によって、内容を次の形へ移せる小さな導線を検討する。候補:

- TempNest から NoteNest の新規ノートへ昇格
- ChatNest の発言を IdeaNest カードとして保存
- IdeaNest カードを NoteNest へ展開
- NoteNest の選択範囲を ChatNest へ引用

自動連携や常時同期は行わない。各操作は、利用者が明示的に選んだ場合だけ実行する。想定する思考の流れ:

> 断片 → 発想 → 対話 → 正式な記録

### 4.3 NestSuite ホーム・振り返り画面

起動時の前回タブ復元は維持する。その上で、任意に表示できる軽量なホームまたは振り返り画面を検討する。候補表示:

- 今日更新したファイル
- 最近使った Workspace
- 未整理の IdeaNest カード
- TempNest に残る内容
- TODO やマーカーの残存状況
- 復元失敗や未保存状態などの現在地

複雑なダッシュボードにはしない。情報を増やしすぎず、次の操作を選べる入口とする。

### 4.4 成果物への変換

NestSuite 内に蓄積した内容を、他者へ共有できる形へ変換することを検討する。候補:

- NoteNest を議事録形式で出力
- IdeaNest カードをアイデア一覧として出力
- ChatNest を検討経緯として出力
- TODO やマーカーのみを一覧出力
- 選択した複数 Workspace の内容を 1 つの Markdown へまとめる
- 共有用 Markdown として出力

自由度の高いテンプレート機構は避け、用途が明確な出力を優先する。

### 4.5 軽い楽しさと愛着

派手な装飾ではなく、丁寧に作られていると感じられる要素を検討する。候補:

- Workspace ごとの控えめな色やアイコン
- 空状態の短い案内
- 保存・コピー・昇格時の自然なフィードバック
- 初回起動時の短い案内
- NestSuite と各 Nest の関係が分かるヘルプ
- 最近使った Nest の小さな表示

アニメーションや装飾を目的化しない。目標とする印象:

> 自分の思考を預けてもよい

L23（v2.18.1）で、NoteNest のノートブック・ノート・タスク・マーカー各一覧が空の場合に、実際に存在する操作へだけ誘導する短い次操作ガイドを追加した。「次に行うことが分かる」方向性の最初の実装であり、今回は NoteNest のみを対象とした。他 Workspace（IdeaNest・ChatNest・TempNest）への横展開は、この実装の実利用を確認してから個別に判断する。

## 5. 優先候補

魅力向上の最初の具体候補として、次を優先する。

### 第1候補（v2.18.0 で実装済み）

**TN-3: TempNest スロット本文の NoteNest 新規ノートへの昇格**

理由:

- 実装範囲を小さくできる
- 保存形式変更が不要
- NestSuite 内で思考が育つ体験を分かりやすく示せる
- TempNest の断片を正式な記録へ進められる
- 自動連携ではなく明示操作で実現できる
- 既存 backlog を利用できる

この体験を次の言葉で表す。

> 断片を正式なノートへ育てる

TN-3 は v2.18.0 で実装し、この最初の導線を実現した（詳細は `docs/release-notes.md` v2.18.0 参照）。
実装は新規 NoteNest タブへの転送のみであり、既存タブへの投入・IdeaNest / ChatNest への転送・
自動連携は含まない。この 1 件の実施結果だけを根拠に、他の Workspace 間転送を自動的に次の
実装対象とはしない。実利用の様子を踏まえたうえで、第3候補以降の着手要否を個別に判断する。

### 第2候補

軽量な振り返りまたは NestSuite ホーム。最初は次の既存情報だけを利用する。

- 最近更新
- 未整理
- TODO あり
- しばらく未使用

高度な分析や推薦は行わない。

### 第3候補

IdeaNest または ChatNest から別 Workspace への昇格。TempNest から NoteNest への導線を実装・評価した後に検討する。

### 第4候補

共有用 Markdown 等の成果物出力。内部で考えた内容を、業務上の成果や共有物へ変換する。

## 6. 避ける方向

魅力向上を理由として、次を安易に導入しない。

- AI 機能
- 自動タグ付け
- 自動関連付け
- 自動推薦
- 複雑なダッシュボード
- プラグイン機構
- 多数の新規 Workspace
- クラウド同期
- 外部 API 連携
- 高機能なプロジェクト管理
- 自動ワークフロー
- 常時動作する Workspace 間連携
- 保存形式の大規模変更
- 外部依存追加

これらは目立つ一方、NestSuite の軽さ、閉域適性、認知負荷軽減方針を損なう可能性が高い。

---

## 7. 実装時の原則

魅力向上に関する機能を実装する場合も、次を守る。

- 1 version = 1 purpose
- 小さな導線から始める
- 自動処理より明示操作を優先する
- 元データを勝手に削除・変更しない
- 転送後の削除は確認を伴う
- Workspace 間の責務を混在させない
- 保存形式変更を避ける
- 現行 schema を維持する
- session 形式を安易に変更しない
- 単一 EXE 志向を維持する
- 外部依存を追加しない
- 機能追加後も認知負荷が増えていないか確認する
- 利用者が機能を使わなくても既存体験を損なわない

---

## 8. 今後の扱い

この文書を根拠として、複数機能を一括実装しない。個別の実装は、通常の backlog 項目として扱う。実装前に、次を個別に確認する。

- 利用場面
- 操作導線
- 元データの扱い
- 失敗時の挙動
- 保存形式への影響
- Workspace 責務
- 認知負荷
- テスト方法

この文書は、NestSuite の今後の魅力向上を検討する際の方向性として維持する。
