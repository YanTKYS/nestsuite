# 対応後レビュー（R-1〜R-8 / v2.16.6〜v2.16.16）

## 総評

**8 件すべてが初回レビューの意図どおり、かつ推奨実装順（R-1 → R-2/R-3/R-4 → R-5 → R-6 → R-7 → R-8）どおりに消化されています。** 各対応は保存形式・session 形式・schema・wrapper を一切変更せず、初回レビューの「局所修正で塞がる」という見立てを裏切っていません。特に評価できるのは次の 3 点です。

1. **R-1 の意味論（.bak = 最後の手動保存）が、コード・文言・テストの 3 層で一貫している。** `createBackup` フラグが 3 FileService すべてに default `true` で通り、自動保存経路のみ `false`。L20 の案内文言「最後に手動保存した時点の内容」もこの意味論を正確に反映しています。
2. **TD-65 の持ち越しと TD-66 の随時保存が正しく協調している。** `_pendingSessionRestoreEntries` は `TryRestoreSession` 冒頭で設定され（`restoredCount == 0` の早期 return より前）、以降のすべての `SaveSession`（随時保存・OnClosing とも）が同じフィールドを経由するため、保存契機が増えても持ち越しは壊れません。復元中は `_isRestoringSession` で随時保存を抑止し、復元完了後に 1 回だけ保存する設計も、中途半端な上書きを構造的に防いでいます。
3. **懸念していた「ファイル指定起動が pending entry を消す」経路は存在しない。** v1.18.2 以降、引数指定起動でも `TryRestoreSession` が先に走るため、pending entry は読み込まれた上で `RegisterLoadedTab` の随時保存に引き継がれます。

新たなデータ喪失経路は見つかりませんでした。見つかったのは「案内の質」レベルの残課題 3 件（後述）です。

## 新たに見つかったリスク

いずれもデータ喪失ではなく「案内・UX」レベルです。

1. **【中】pending entry の恒久 nag に解除手段がない**（TD-65 の副作用）。恒久的に削除されたファイルが session に残ると、**起動のたびに復元失敗ダイアログが永遠に出続け**、ユーザー側に「再試行をやめる」手段がない（session.json の手動削除以外）。初回レビュー R-3 の「1 起動分持ち越し」を超えて無期限にしたことの代償。対策は小さい: ダイアログに「次回から再試行しない」選択肢を足すか、FileNotFound のみ N 回連続失敗で自動除外する（どちらも session 形式変更なしで可能 — 持ち越さないだけ）。
2. **【小】複数件通知で .bak 案内が落ちる**。`NotifyRestoreFailures` と `MultipleOpenFailureMessageBuilder` はともに理由の 1 行目のみを使うため、InvalidFormat の「.bak から復元できる可能性」（L20）が複数件表示では届かない。単体で開き直せば全文が出るので実害は薄いが、「破損が疑われるファイルは単体で開き直すと詳しい案内が出ます」の共通末尾 1 行を将来足す価値はある。
3. **【小・既知として記録推奨】ActiveFilePath の鮮度非対称**。TD-66 はアクティブタブ変更のみを対象外にしたため、クラッシュ復元時「タブ構成は最新・フォーカス位置だけ古い」ことがある。release notes には明記済みだが、design docs 側にも一文あると将来の混乱を防げる。

また、**R-14 のドリフト面が TD-65 で現実に広がった**点を指摘します。`CreateSessionState` の pending merge が `filePaths` と `tabStates` へ**並行 append** しており、これはまさに R-14 が警告した二重導出パターンです。現在はテストで守られており安全ですが、恒久対策（FilePaths を Tabs から導出）が数行で済む状況です。
