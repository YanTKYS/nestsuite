- 関連: `docs/planning/attractiveness-review-2026.md` §10 AT-1 / §12 v2.18.16欄 / 方向性文書 4.3「第2候補」

## 0. 結論サマリー

**条件付き採用**。専用パネルは作らず、**通常タブの復元・作成が一切なかった起動に限り、TempNest上部（AT-5一行ガイドと同じ領域）へ「続きから」行を表示する**（後述の案B）。表示内容は既存の recent files 上位3件（ファイル名のみ・クリックで既存 open 経路により開く）と、下書きが保持されている場合の受動的な1行案内に限定する。内容プレビューは表示しない。保存形式・UI settings の変更なし。起動時の追加ファイルI/Oなし。候補が0件の場合は現行の AT-5 一行ガイドをそのまま表示する（AT-5 は削除しない）。

## 1. 現行実装の確認結果（設計の根拠）

設計判断の前提として、リポジトリの現行実装を確認した。以下は本レビューの判断が依拠する事実である。

### 1.1 起動・復元（`NestSuiteShellWindow` コンストラクター）

起動処理の順序は次のとおり（`NestSuiteShellWindow.xaml.cs`）。

1. UI settings 読込（破損時は退避+既定値、M19）・テーマ適用
2. `UpdateRecentFilesMenu()` — **この時点で recent files JSON は既に1回読まれている**
3. `TabStrip.ItemsSource = _tabs` 設定後、AT-5 の `WireGettingStartedHintTracking()`（`_tabs.CollectionChanged` を1箇所購読）
4. TempNest 固定タブ（左端・常設）を追加
5. `TryRestoreSession()` — session 復元。復元失敗 entry は `_pendingSessionRestoreEntries` へ保持し `NotifyRestoreFailures()` が**起動時に1回ダイアログ通知**（FileNotFound を含む場合は再試行解除の Yes/No 統合ダイアログ、SH-34）
6. 復元保留 entry があれば AT-5 ラッチを立てる
7. 復元なし・起動引数なしの場合のみ `ActivateTab(tempTab)` — **「再開対象が何もない起動」は TempNest がアクティブになる分岐として既にコードで一意に判別できる**（`!restoredSession && NestSuiteStartupTabPolicy.ShouldCreateInitialTab(initialFilePath)`）
8. `RestoreDraftsAtStartup()` — 下書きがあれば**起動時に1回 Yes/No/Cancel ダイアログ**（はい=無題タブ復元 / いいえ=破棄 / キャンセル=保持して次回再確認）
9. 自動保存タイマー開始

前回アクティブタブは session の `ActiveFilePath` から復元成功時に再選択される。起動引数・ファイル関連付け・pipe 転送（`OpenFileFromPipe`）・recent files メニューは、いずれも最終的に `ShellFileOpenPlanner.Plan` → `LoadWorkspaceFileAt` / `ActivateExistingTabForOpen` の既存 open 経路に合流し、タブ追加は必ず `_tabs.Add` を通る（AT-5 がこの性質を既に利用済み）。

### 1.2 最近使ったファイル（`NestSuiteRecentFilesService`）

- 保存先: `%APPDATA%\NoteNest\nestsuite-recent-files.json`。**パス文字列の単純な JSON 配列のみ**（Workspace種別・日時・メタデータなし）
- 上限 10 件・MRU順（先頭が最新）・追加時に同一パスを除去してから先頭挿入（重複なし）
- メニュー表示: ファイル名（`Path.GetFileName`）+ ToolTip にフルパス。**表示時にファイル存在確認はしない**。クリック時に不存在なら エラーダイアログ + 一覧から削除
- Workspace 種別: legacy 拡張子（`.notenest`/`.ideanest`/`.chatnest`）は拡張子から判定可能だが、**`.nestsuite` はファイル内容（wrapper）を読まないと種別が決まらない**（`NestSuiteOpenFilePolicy`/`NestSuiteTabFactory`）

### 1.3 session（`NestSuiteSessionState` / `NestSuiteSessionStateService`）

- 保存先: `%APPDATA%\NoteNest\nestsuite-session.json`。`FilePaths[]`・`ActiveFilePath`・`Tabs[]{FilePath, WorkspaceKind, IsPinned}`
- 未保存タブ（FilePath == null）は session に含まれない
- `Tabs[].WorkspaceKind` は**保存時のUI表示ヒント**であり、復元判定の信頼ソースではない（TD-68 の明示コメント: 「将来の選択的復元 UI 等で、ファイルを読み込む前に種別を表示するため」）。AT-1 が種別表示に使える既存フィールドはここにしかない
- タブ変更ごとの随時保存（TD-66）+ 終了時保存。破損時は `.corrupt` 退避（TD-65）

### 1.4 draft（`DraftStore` / `NestSuiteShellWindow.DraftRecovery`)

- `draftFormatVersion 1.0`（sidecar）。tabId 単位のファイル。**一覧（`ListDraftFiles`）は件数取得だけならファイル内容を読まない**が、Workspace 種別・内容は wrapper を読まないと分からない
- 起動時 `RestoreDraftsAtStartup()` が件数付き Yes/No/Cancel ダイアログで復元・破棄・保持を確認する。**「キャンセル=保持」を選ぶと、その起動中に draft を再確認する導線は存在しない**（次回起動時に再確認）
- 破損 draft は quarantine 退避。SH-37 サマリーが「開いているタブの自動保存分を除いた候補件数」を `ShellStateSummaryCalculator.CountDraftRecoveryCandidates` で算出する既存実装がある

### 1.5 TempNest / AT-5

- TempNest は左端固定タブ・2×2 の4スロット・`tempnest.json`。空判定は `TempNestSlotViewModel.IsEmpty`（`IsNullOrEmpty`、空白のみは「空でない」）
- AT-5 一行ガイドは `TempNestWorkspaceView.xaml` の Grid Row 0（スロットの上）に配置された `TextBlock` 1個。表示条件は `ShouldShowGettingStartedHint`（4スロット全空 かつ 同一起動中に再開対象・入力が一度もない）。抑止は Shell からの `MarkGettingStartedHintDismissed()`（単発ラッチ・永続化なし）
- **AT-1 の初期仮説が置く「TempNest上部」は、AT-5 が既に確保している表示領域と同一**

### 1.6 SH-37 状態サマリー

メニューから明示的に開く読み取り専用ダイアログ。開いているタブ数・未保存数・復元保留数・draft候補数・TempNest非空スロット数の**件数のみ**。attractiveness-review-2026.md は AT-1 を「SH-37（件数）に中身と入口を足すもの」と位置づけている。

### 1.7 detached window

`OnClosing` で detached window は先にすべて閉じられ、session に detached レイアウトは保存されない。復元は常に Shell のタブとして行われる。

## 2. 問い1: スタートパネルは必要か（案A〜E比較）

| 観点 | 案A 専用パネル | 案B TempNest上部埋め込み | 案C 既存メニュー強化 | 案D 前回アクティブ復元の改善のみ | 案E 何もしない |
|------|------|------|------|------|------|
| 初見の分かりやすさ | 高いが「もう1画面」が増える | 高い（起動直後の画面に自然に載る） | 低い（メニューを開くまで見えない） | 変化なし | 変化なし |
| 再訪のしやすさ | 高い | 高い | 中（現状の弱点=メニューが2階層下のまま） | 低（sessionが空なら効果なし） | 低 |
| 認知負荷 | 中〜高（パネルの寿命・閉じ方の学習が必要） | 低（既存画面の1領域） | 低 | 最低 | 最低 |
| UI占有 | 大 | 小（AT-5行の置換） | なし | なし | なし |
| 実装量 | 大（新View・ライフサイクル・フォーカス管理） | 小〜中 | 小 | 小 | ゼロ |
| 起動速度 | 中リスク | 影響なし（既読情報のみ） | 影響なし | 影響なし | 影響なし |
| 保存形式への影響 | なし〜あり（表示設定が欲しくなりがち） | なし | なし | なし | なし |
| 誤操作リスク | 中（起動時フォーカス簒奪） | 低 | 低 | 低 | なし |
| 情報露出 | 中（起動即・大面積で一覧表示） | 低〜中（3件・ファイル名のみに絞れる） | 低（明示操作後のみ） | なし | なし |
| 将来保守 | 恒久ホーム画面へ肥大化する主リスク | 領域が物理的に小さく肥大化しにくい | 低 | 低 | 低 |
| AT-5との重複 | 排他制御が別途必要 | 同一領域なので排他が自然に定義できる | なし | なし | なし |
| 既存メニューとの重複 | recent メニューと二重 | 「上位3件のみ・全表示はメニュー」で役割分担可 | 同一物の改善 | なし | なし |

**判断**:

- **案E は棄却**。session 復元は「前回開いたまま終了したタブ」しか救えない。NestSuite の推奨ワークフロー（保存して閉じる）に従うほど session が空になり、再開の手がかりが「ファイル > 最近使ったファイル」の2階層下だけになる。これが §3-1「再訪の入口がない」の実体であり、AT-5（初回向け文言）では答えられない。
- **案D は棄却**。前回アクティブ復元は既に実装済み（session の `ActiveFilePath`）。改善余地は「sessionが空のとき」にあり、案Dはその場合に何も提供できない。
- **案C は単独では不足**。メニュー改善は「探しに行ける人」しか救わない。起動直後に受動的に見えることが AT-1 の価値。ただし「すべて表示」の受け皿として既存メニューを維持する形で案Bに包含する。
- **案A は棄却**。読み取り専用と言いつつ独立パネルはライフサイクル（いつ出す・いつ消す・フォーカスどこ）の設計コストが大きく、リスク表（§13）の「恒久管理画面へ肥大化」の主経路になる。初期仮説も「専用の大きなホーム画面は作らない」としており、調査結果はこれを支持する。
- **推奨: 案B（TempNest上部埋め込み）**。根拠: (1) 「再開対象なし起動」は現行コードで既に一意な分岐として判別でき、その分岐の着地先が TempNest である。(2) AT-5 が同じ領域・同じ表示条件系・同じ Shell→TempNest 連携パターン（プロパティ+ラッチ）を確立済みで、そのまま拡張できる。(3) 領域が物理的に小さいため「増やしすぎない」制約が構造的に守られる。

## 3. 問い2: いつ表示するか

**表示条件（一文）: 起動時に通常タブの復元・作成が一切行われなかった場合（= 現行コードで TempNest がアクティブ化される起動分岐）に限り表示し、同一起動中に通常タブが1枚でも追加されたら以後表示しない。**

これは AT-5 の表示条件の上位互換であり、判定材料はコンストラクターが既に持つ情報（`restoredSession`・`initialFilePath`・`_tabs.CollectionChanged`）だけで足りる。新たな永続フラグは追加しない。

各ケースの判断:

| ケース | 表示 | 理由 |
|------|------|------|
| 初回相当・TempNestも空・recent 0件 | しない（AT-5 を表示） | 続きが存在しない。AT-5 の領分 |
| TempNestに内容がある（通常タブなし） | する | TempNest の中身は画面に見えているため項目としては出さない（§4） |
| session復元成功 | しない | タブが既に開いている。二重表示（§6）を避ける |
| sessionの一部だけ復元成功 | しない | 1枚でも通常タブがあれば「続き」は目の前にある。失敗分は既存の起動時ダイアログが通知済み |
| session復元失敗（全滅・復元0件） | する | `restoredSession == false` に該当。ただし失敗通知自体は既存ダイアログの責務のまま |
| session復元保留 entry あり（タブなし） | する（ただしパネルは recent files のみ表示） | 保留 entry の通知・再試行解除は既存の SH-34 統合ダイアログの責務。パネルで二重に扱わない |
| draft あり | する（draft は受動的1行のみ、§7） | draft 復元ダイアログが先に出る。「キャンセル=保持」後の控えめな備忘だけ担う |
| 起動引数あり | しない | 開くファイルを利用者が既に指定している |
| ファイル関連付け起動 | しない | 同上（initialFilePath 経路） |
| pipe転送 | 既存ウィンドウ側の表示状態を変えない（転送でタブが増えれば自動非表示） | `OpenFileFromPipe` は `_tabs.Add` を通るためラッチが自然に働く |
| recent file から起動（メニュー） | タブ追加時点で非表示化 | 同上 |
| 通常Workspaceを新規作成後 | 非表示化（以後この起動中は再表示しない） | AT-5 と同じ単発ラッチ |
| すべての通常タブを閉じた後 | 再表示しない | ラッチは戻さない。再表示の複雑さ（閉じた直後のファイルが「続きから」に出る循環）を避ける。フェーズ3で実機評価 |
| 利用中に明示的に再表示 | 提供しない（§15） | 恒久化の入口になるため |

**AT-5 との排他**: 同一領域で「続きから候補が1件以上 → AT-1 行を表示 / 0件 → 現行 AT-5 文言を表示」。両方同時には出さない。AT-5 の判定・文言・テストは変更せず、表示の切り替えだけを上位で行う（§20）。

## 4. 問い3: 何を表示するか

| 情報 | 区分 | 理由 |
|------|------|------|
| recent files 上位3件（ファイル名） | **初期表示に必須** | AT-1 の中核。既読の JSON から取得、I/O 追加なし |
| 各項目のフルパス | 補助表示（ToolTip のみ） | 既存 recent メニューと同じ形。常時表示は情報露出・幅の観点で不採用 |
| 保持中 draft の存在（件数のみの1行） | 初期表示（存在する場合のみ） | 「キャンセル=保持」後の唯一の備忘。操作ボタンは持たない（§7） |
| Workspace 種別（アイコン・ラベル） | **表示しない（フェーズ1）** | recent files JSON に種別がなく、`.nestsuite` は内容を読まないと判定できない（§1.2）。種別表示のためだけに起動時ファイルI/Oを増やすことは §10 の方針に反する。フェーズ2で session の `Tabs[].WorkspaceKind` ヒント（TD-68 が予約した用途そのもの）を「session に載っている項目に限り」流用する案を再評価する |
| 前回アクティブファイル（独立項目として） | 表示しない | session 復元成功時はタブとして復元済み（パネル非表示）。session が空のときは `ActiveFilePath` も空であり、実質 recent[0] と一致する。独立枠は重複（§6） |
| session 復元済み・失敗・保留・破損・重複・再試行 | 表示しない | すべて既存の起動時ダイアログ（SH-31/SH-34/TD-65系）の責務。パネルへ再掲すると責務が二重になる |
| 最終利用日時・親フォルダ | 表示しない | recent files に日時がなく、取得にはファイルstatが必要。表示価値も3件なら低い |
| ファイル不存在マーク | 表示しない | §9 参照 |
| TempNest 残存（件数・プレビュー・開くボタン） | 表示しない | パネル自体が TempNest 上部にあり、スロットの中身は直下に見えている。項目化は同語反復 |
| draft の種別・名前・内容プレビュー | 表示しない | 種別・内容は wrapper を読まないと不明（§1.4）。露出リスクも高い |
| 各カード・ノート等の内容プレビュー | 表示しない | §8 参照 |
| 新規作成ボタン群（NoteNest/IdeaNest/ChatNest） | 表示しない | タブバーの「＋」が既にあり、AT-5 文言が案内済み。重複導線 |

初期仮説からの修正点: 仮説は「1. 未保存draftまたは復元保留 / 2. TempNest残存 / 3. recent上位3件」の3種を置いていたが、調査の結果 (a) TempNest残存はパネルの設置場所自体が TempNest であるため項目として不要、(b) 復元保留は既存の SH-34 統合ダイアログと責務が衝突するため不掲載、(c) draft は既存ダイアログとの重複を避け「操作を持たない備忘1行」へ縮小、が妥当と判断した。

## 5. 問い4: 優先順位

表示順（上から）:

1. **draft 備忘行**（存在する場合のみ）— 唯一「失うと戻せない」情報であり最優先。ただし復元操作は次回起動のダイアログへ委ねる
2. **recent files 上位3件**（MRU順そのまま）— 「すぐ再開できる」代表。並び替え・独自スコアリングはしない

「sessionで前回開いていたファイル」「前回アクティブ」はパネルが出る状況（=復元なし起動）では実体が recent files と重なるため独立項目にしない。「新規作成」は既存導線（＋ボタン）に委ねる。重複を排した結果、優先順位問題はほぼ消滅する — これは表示条件を「復元なし起動のみ」に絞ったことの副次効果であり、この設計の簡潔さの根拠でもある。

## 6. 問い5: session 自動復元との重複

- **自動復元は現行のまま維持する**(復元の選択式化は行わない。TD-59〜TD-70 で安定化させた復元系への大変更であり、AT-1 の目的に対して過大)
- **session 復元が1件でも成功した起動ではパネルを表示しない**。これにより「復元済みタブとパネルの二重表示」は構造的に発生しない
- 復元済みタブのサマリ表示・前回アクティブ強調は不採用(前回アクティブは復元時に既に選択される)

## 7. 問い6: draft との関係

- draft 復元・破棄の主導線は現行の起動時ダイアログのまま変更しない
- パネルには「保存されていない下書きが N 件あります（次回起動時に確認できます）」の**受動的な1行のみ**。復元ボタン・破棄ボタン・内容プレビュー・種別表示は置かない
- 破棄操作をパネルへ置かない理由: 「続きから」画面に破壊的操作を並べない原則（依頼文の方針に同意）。誤クリック1回で不可逆になる操作は確認ダイアログ付きの既存フローに限定する
- 件数は起動時に `RestoreDraftsAtStartup` が既に列挙した結果を使い回す（再列挙しない）。SH-37 の候補算出（開いているタブ分を除外）と同じ計算を共有できる

## 8. 問い7: TempNest との関係

§2 のとおり「TempNest 上部へ小さく埋め込む」を採用。他候補の評価:

- **上に重ねる（オーバーレイ）**: 不採用。スロットを隠し、閉じる操作が必要になる
- **一時的に置き換える**: 不採用。TempNest へすぐ書きたい利用者を1操作遅らせる
- **別固定タブ「開始」**: 不採用。固定タブが2枚になり、タブの意味（=作業単位）を壊す。恒久ホーム化の入口
- **TempNest をスタート画面として拡張**: 部分的に採用が実態。ただし「一時メモ＋（復元なし起動時のみの）再開入口」であり、TempNest の保存形式・スロット機能・軽さは一切変えない。2×2 スロットの視認性への影響は AT-5 と同じ Row 0 の高さ増分（数行分）のみ

## 9. 問い8: 情報プレビュー

**内容プレビューは全面的に不採用**。表示するのはファイル名（拡張子含む）と、ToolTip 限定のフルパスのみ。

- 自治体・閉域環境での画面共有・離席・スクリーンショットを考慮すると、起動直後に自動表示される領域に本文・ノート名・発言・カード数等を出すべきでない
- ファイル名自体が機微になり得る点は残るが、これは既存の recent files メニュー・タブ見出し・タイトルバーと同水準であり、新たな露出面を増やさない（既存と同じ情報を既存より1階層浅く出すだけ）
- NoteNest 先頭ノート名・IdeaNest カード数・ChatNest 最終発言・draft 先頭数十文字は、いずれも取得にファイル読込が必要で §10 にも反するため二重に不採用

## 10. 問い9: ファイル不存在・破損

- **表示時の存在確認はしない**（グレーアウト・自動削除・不存在マークなし）。既存 recent メニューが表示時に検査しない方針と揃える。3件の `File.Exists` は安価だが、ネットワークドライブ上のパスでは同期 stat がブロックし得るため、起動描画経路では行わないのが安全
- クリック時は既存の `ShellFileOpenPlanner.Plan` に完全委譲: 不存在 → 既存のエラーダイアログ + recent から削除（パネル表示も更新）。種別判定失敗 → 既存の理由別メッセージ。`.bak` 復元導線・再試行表示は追加しない
- パネル独自のファイル検査・schema 解析は行わない

## 11. 問い10: 起動速度

- **追加I/Oゼロ**を設計制約とする。recent files はコンストラクターが `UpdateRecentFilesMenu()` で既に読んでおり、その結果を使い回す（フィールド保持または再利用可能な形へ小さく整理）。draft 件数は `RestoreDraftsAtStartup` の列挙結果を使い回す
- ファイル内容の読込による種別判定・プレビュー生成・存在確認・遅延読込・段階更新は、いずれも行わない（行う情報を表示しないため不要になる — 表示の絞り込みが性能対策を兼ねる）

## 26. 問い25: テスト戦略

- **純粋ロジック（新規 Policy/Builder クラス、WPF 非依存）**: 表示可否判定（復元あり/なし・引数あり/なし・タブ追加後・候補0件時の AT-5 フォールバック）と項目構築（recent リストから上位3件・draft 件数行の有無）を、recent リスト・draft 件数を引数注入して単体テスト。実ファイル I/O・実 `%APPDATA%` に依存させない（`MakeEmptyTempNestViewModel` と同じ隔離方針）
- **表示条件**: 初回相当（候補0）→ AT-5 / recent のみ → 3件 / draft のみ → 備忘行のみ / 両方 / 復元成功 → 非表示 / 引数起動 → 非表示 / 通常タブ追加後 → 非表示、を Policy テストで網羅
- **項目**: MRU 順維持（並べ替えないことの確認）・4件以上あるとき3件に切る・0〜2件時の縮退・重複は recent 側で既に排除済みであることへの依存を明記
- **操作**: クリック→既存 open 経路が呼ばれること（デリゲート注入で確認）。開く失敗・キャンセルはパネル層でテストせず既存 Planner テストに委ねる
- **保存形式不変**: 既存の schema/session/draft/recent 回帰テスト群（`IdeaNestFormatSchemaRegressionTests` 等）が変更なしで通ること。UI settings へのフィールド追加がないことは docs-contract/静的テストで確認
- **静的 UI 契約**: AutomationId・DynamicResource のみ使用・新規固定色なし・AT-5 TextBlock の残存、を AT-5/ID-6 と同形の小さな静的テストで確認（XAML 全文固定・座標固定はしない）
- **UI smoke**: 既存 smoke が起動分岐（復元なし起動）を通るため、パネル追加後も smoke が green であることを確認。smoke へのシナリオ追加は最小限
- Shell コンストラクターの起動分岐そのもの（Window 構築が必要）は既存方針どおり単体テスト対象外とし、判定ロジックを Policy へ切り出すことでカバーする

## 28. 問い27: 最終判断

- **採用判断**: **条件付き採用**（フェーズ1の最小形に限定。専用パネル・恒久表示・プレビュー・操作拡張は不採用）
- **推奨UI形態**: **TempNest埋め込み**（TempNest 上部・AT-5 領域の排他利用）
- **初期表示条件**: 起動時に通常タブの復元・作成が一切なかった場合に限り表示し、同一起動中に通常タブが追加されたら以後表示しない
- **初期表示項目**（最大4項目）: (1) recent files 上位3件のファイル名リンク（最大3）、(2) 保持中 draft の件数備忘行（存在時のみ・操作なし）
- **表示しない情報**: 内容プレビュー全般・Workspace種別・日時・親フォルダ・フルパス常時表示・session復元状態・TempNest残存項目・新規作成ボタン・不存在マーク
- **保存形式変更**: **なし**（session・draft・recent files・TempNest・UI settings・schema すべて不変）
- **設計レビュー追加要否**: **通常エンジニアで実装可能**（本文書の範囲内であれば追加レビュー不要。範囲を超える拡張 — 種別表示・手動再表示・操作追加 — を行う場合はフェーズ2として再レビュー）
- **推奨実装単位**: 1 version（採番候補 SH-40）= フェーズ1全体（表示条件 + recent 3件 + draft 備忘行 + AT-5 フォールバック + テスト一式）。これで 1 version = 1 purpose を満たす
- **実機確認後の判断事項**（5件以内): (1) TempNest 入力後もパネルを維持するか消すか、(2) 種別表示（session ヒント流用）の要否、(3) draft 行から確認フローへの導線の要否、(4) 3件という件数の妥当性、(5) 「今後表示しない」設定の要否
