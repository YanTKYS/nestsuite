# LT-9 フェーズ2設計レビュー

## 総評

**フェーズ2は「設計は固められるが、着手はまだ」が結論です。** SH-34（v2.16.21）が出荷された直後であり、review4 自身が「フェーズ1の実績を見てから」と定義した観察期間がまだゼロです。さらに重要なのは、フェーズ2が解消する残存問題の実害を冷静に見積もると、いずれも軽微だということです。

- **all-or-nothing 解除の害**: 「はい」で一時未接続のファイルまで外れても、失われるのは自動再試行だけです。ファイルもデータも消えず、最近使ったファイルや Open ダイアログから 1 回開き直せば session に戻ります
- **InvalidFormat 等の恒久 nag の害**: 起動ごとに OK ダイアログ 1 枚。間接経路（修復 / .bak 復元 / 削除して FileNotFound 化）は v2.16.22 でユーザーガイドに文書化済み

一方でフェーズ2のコストは、この session/復元系で**初のカスタム Window**（これまで全て MessageBox で済ませてきた）+ チェックボックス一覧 + 解除対象拡張の安全設計であり、v2.16.x 系で最大の UI 追加になります。実害軽微 × コスト最大の組み合わせで、トリガーなしに今進める根拠がありません。

ただし本レビューで設計判断は出し切れます。**着手条件・表示条件・解除範囲・API 形を今回確定して backlog に固定し、トリガーが観測された時点で実装プロンプトに落とせる状態**にしておくことを推奨します。

また、レビュー中に**実装時の技術的制約を 1 件発見**しました: 復元失敗通知はコンストラクター内（メインウィンドウ `Show()` 前）で発火します。`MessageBox.Show(owner, ...)` は未表示 owner でも動作しますが（現行実績あり）、**カスタム `Window.ShowDialog()` で `Owner` に未表示の Window を設定すると `InvalidOperationException` が発生します**。フェーズ2実装時は「Owner を設定せず `CenterScreen` で出す」か「通知タイミングを `Loaded` 後へ移す」のどちらかが必要で、後者は現行挙動の変更を伴うため、前者を推奨します。この制約は実装プロンプト作成時に必ず含めるべきです。

## フェーズ2を今実装すべきか

**まだ実装しない。** backlog に留め、以下のいずれかをトリガーとして着手する:

1. all-or-nothing 解除で「外したくないものまで外れた / 外せなかった」という実利用フィードバック
2. InvalidFormat / AccessDenied 等の恒久 nag が、文書化済みの間接経路では不十分という実利用フィードバック
3. SH-35（解除対象拡張）を進める判断がされたとき — 拡張を安全にやるには個別選択 UI がほぼ必須であり、現行の一括 Yes/No に拡張を載せる方が危険なため、SH-35 実装 = フェーズ2実装と同義

認知負荷の観点でも、正常起動 0 枚・失敗時 1 枚という現状は既に最適点に近く、UI を先回りで増やす理由がありません。

## 推奨する UI 表示条件

実装する際の条件は、候補 A〜E のどれでもなく、**件数ベースのハイブリッド**を推奨します:

- **失敗 entry が 2 件以上 → フェーズ2ダイアログ**（全失敗種別を一覧表示）
- **失敗 entry が 1 件 → 現行 SH-34 の挙動を維持**（FileNotFound なら Yes/No、それ以外なら OK 通知）

理由: 個別選択の価値は 2 件以上で初めて発生します。1 件に対してチェックボックス一覧を出すのは過剰で、最頻ケース（失敗 1 件）を重くします。種別ベースの条件（A/B）より件数ベースの方が規則として単純で、利用者にも説明しやすい。C は retryCount が必要で禁止事項に抵触、D は失敗一覧が起動時の一時情報であることと発見性の悪さから不採用（review4 の方針維持）。

## 推奨する最小 UI

- タイトル: 「前回復元できなかったファイル」
- 説明文: 「チェックしたファイルは、次回から復元対象から外せます。外してもファイル自体は削除されません。」
- 一覧: 1 行 = チェックボックス + ファイル名 + 理由（1 行文言）。フルパスは ToolTip
- フッター: InvalidFormat を含む場合のみ .bak 誘導 1 行（既存 `MultipleFailuresBakDetailHint` を流用）
- ボタン: 「選択したファイルを次回から再試行しない」（未選択時は無効化）/「今回は何もしない」
- Esc / ✕ = 「何もしない」と同義

## 表示項目・文言

- **チェックボックスは必要**（個別選択がフェーズ2の存在理由そのもの）
- **初期値は全て未チェック**。安全側（= 従来どおり再試行）に倒す
- **「すべて選択」ボタンは付けない**。一括解除したいケースは稀で、誤操作リスクが上回る
- **ファイル名表示で足りる**。フルパスは ToolTip。同名ファイルが複数ある場合のみ判別が必要になるが、ToolTip で解決できる範囲
- **種別列は省略を推奨**。legacy 拡張子ならファイル名から自明、`.nestsuite` の失敗 entry は種別判定に失敗した結果であることが多く「不明」表示が並ぶだけ。情報量ほぼゼロの列にプラミング（`Tabs[].WorkspaceKind` ヒントとの突合）を追加する価値がない。将来入れる場合は Tabs[] ヒント + null →「不明」（TD-68 の方針どおり）

## 解除対象の範囲

**フェーズ2では全失敗種別を解除可能にすることを推奨**（SH-35 の実装部分を吸収）。

根拠は「解除は非破壊」であること: entry を外してもファイルも `.bak` も残り、失うのは自動再試行の通知だけです。個別選択 + 明示文言があれば、種別で解除可否を分ける複雑さの方が害になります。ただし種別ごとの注意書きは必要:

- **InvalidFormat**: 行またはフッターに「外す前に、単体で開き直して .bak からの復元を確認してください」
- **AccessDenied**: 「権限やネットワークの一時的な問題の場合は、外さず再試行を続けてください」
- **SchemaVersionTooNew**: 「新しいバージョンの NestSuite で開ける可能性があります」

これらは静的な注意文とし、**チェック時の追加確認ダイアログは出さない**（ダイアログを減らすためのフェーズ2でダイアログを増やさない）。

## 安全側の既定値

- 既定全て未チェック / 「すべて選択」なし / 未選択時は実行ボタン無効
- 「外してもファイル自体は削除されません」を説明文に明記する（「外す」の意味の誤解が最大のリスク）
- 実行ボタン文言は「選択したファイルを次回から再試行しない」— 対象と効果が文言だけで完結する形

## session.json 形式変更の要否

**不要（review4 の判断はフェーズ2でも妥当）。**

- 選択解除に必要な情報は **FilePath 集合だけで足りる**。失敗理由は表示にのみ使い、除外判定には不要
- 重複は既存の `DeduplicatePendingEntries` と session 構築経路で排除済みの前提でよい
- パス比較は既存どおり `NestSuiteOpenFilePolicy.IsSameFile`（大文字小文字無視）に統一する。UI 選択と pending entry の突合もこれを使うこと
- 解除済み履歴の保存は不要。「外した後にやはり必要だった」場合はファイルを開き直せば session に戻る、が既存の意味論であり、履歴がなくても復帰経路が存在する
- 解除対象の種別を session に残す必要もない（理由は起動時再判定で得る方針 = design-decisions §55 のとおり）

## helper / API 設計

```csharp
// 一般化 helper（推奨）
public static IReadOnlyList<SessionRestoreFailure> RemoveEntriesByPath(
    IReadOnlyList<SessionRestoreFailure> pendingRestoreEntries,
    IReadOnlyCollection<string> filePathsToRemove)
```

- **入力は FilePath 集合**を推奨。UI が表示した行の同一性はパスが自然なキーで、record 等価比較はコピー・変換で壊れやすい
- **reason 種別は helper で見ない**。何を選択可能にするかは UI の責務、helper は指名されたパスを外すだけ（責務分離）
- 既存の `RemoveFileNotFoundEntries` は**削除しない**（既存テストが参照）。内部を `RemoveEntriesByPath` への委譲に書き換えるのは可
- 行データの組み立ても UI 非依存に: `SessionRestoreFailuresMessageBuilder` の隣に「行モデル（ファイル名 / 理由文言 / フルパス）を作る」builder を置き、挙動テスト可能にする。カスタム Window の code-behind は「行リストを表示し、チェックされたパス集合を返す」だけの薄さに保つ

## SH-34 との関係

- **1 件失敗時は SH-34 の Yes/No MessageBox を残す**（前述の件数ベース条件）。置き換えは 2 件以上のみ
- `SessionRestoreFailuresMessageBuilder.BuildFailuresMessage` / `ForgetFileNotFoundQuestion` は 1 件経路で引き続き使うため維持
- 片方の経路だけ文言が古くなるドリフトに注意。共通部分（理由 1 行文言・.bak 誘導）は builder / const 共有を維持すること

## .bak 案内との関係

- 行ごとの .bak 案内は**しない**（縦幅と反復の害）。フッターに 1 回、InvalidFormat を含む場合のみ表示（TD-71 の様式を維持）
- InvalidFormat 行がチェックされた場合の追加モーダルは**出さない**。フッター注意文「外す前に単体で開き直して .bak からの復元を確認してください」で足りる
- 「単体で開き直すと .bak 復元案内が出る場合があります」の既存文言は維持する

## 特に知りたい結論への回答

1. **まだ実装しない。** SH-34 直後で観察期間ゼロ、残存する害は軽微かつ文書化済みの回避策あり。着手トリガー 3 条件（all-or-nothing の実害報告 / nag の実害報告 / SH-35 推進判断）を backlog に固定し、成立時に着手する
2. 実装する場合は**失敗 entry 2 件以上のときのみ**フェーズ2ダイアログを表示。1 件のときは現行 SH-34 の挙動を維持（種別ではなく件数で分岐）
3. **含めるべき（フェーズ2実装時）。** 解除は非破壊（ファイル・.bak は残る）であり、個別選択 + 種別別の静的注意文があれば全種別解除可能が一貫する。SH-35 の実装部分はフェーズ2に吸収する
4. **不要。** 選択解除は FilePath 集合で完結し、失敗理由は起動時再判定・履歴保存なしの既存方針（design-decisions §55）のままフェーズ2まで成立する
5. **残す。** 1 件失敗時の Yes/No MessageBox は最軽量の正解であり、フェーズ2ダイアログは複数件時のみの置き換えとする
6. **凍結継続。** フェーズ3の前提（需要・起動性能課題の実測）に変化なし
