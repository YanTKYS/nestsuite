## 1. エグゼクティブサマリー

1. **NestSuite の現在の最大の強みは「壊れない・迷わない・軽い」の三点が実装レベルで裏づけられていることである。** `.bak`・無題 draft・session 復元・破損時退避・復元失敗の解除導線という多層のデータ保護、操作結果のステータス通知、空状態ガイド、テーマ・アクセシビリティ・ショートカット一覧まで、「思考を預けてよい」と感じさせる土台は完成度が高い。これは v2.16 系の保護強化と v2.18 系の体感改善の蓄積による成果であり、同種のローカルメモツールに対する実質的な差別化点である。
2. **最大の弱点は、蓄積した情報が「再び役に立つ」体験がまだ細いことである。** 書く・保存する・守るは強いが、閉じたファイルは「最近使ったファイル 10 件」以外からほぼ再発見できない。横断検索（Ctrl+Shift+F）は**開いているタブのみ**が対象であり、蓄積が増えるほど価値が上がる構造になっていない。方向性文書が掲げる「蓄積した思考を、再発見し、次の形へ進められるツール」のうち、「再発見」の実装が最も遅れている。
3. **4 Workspace 構成は適切であり、再編は不要。** 弱いのは各 Workspace ではなく「Workspace の間」（再訪の入口、閉じたファイルへの到達、成果物への出口）である。
4. **推奨する次の一手は、(1) 入口＝「続きから」スタートパネル、(2) 再発見＝横断検索の対象拡張、(3) 安心＝IdeaNest Undo、(4) 出口＝成果物出力の均質化、(5) 信頼＝利用者向け文書の鮮度修正**の 5 点。いずれも保存形式変更なし・外部依存なしで実現できる。
5. 初回体験には**実装よりも文書の問題**がある。README のスキーマ表記（`1.4.1`）・拡張子の説明（`.nestsuite` 統一が未反映）・ユーザーガイドの起動説明（「無題 NoteNest から始まる」）・既知制約文書（v2.10.0 時点のまま）が現状と食い違っており、最初に読む文書が古いことは「丁寧に作られている」という印象を直接損なう。

## 2. 現在の強み

実装確認に基づく強み（担保している実装・文書を併記）:

| 強み | 根拠 |
|------|------|
| データを失わない安心感 | `.bak`（手動保存時のみ更新）、無題タブの 30 秒 draft 保存 + sidecar、session 復元と失敗 entry の持ち越し・解除導線（SH-34/LT-9 フェーズ1）、破損ファイルの `.corrupt-*` 退避（M19 で ui-settings / recent files まで拡張） |
| 迷いにくさ | 新規作成メニューの説明的表記（「新規 NoteNest — ノートをプロジェクト単位で管理」）、L23 空状態ガイド、無効ボタンの理由付きツールチップ、ショートカット一覧（SH-19）、CH-16 の操作導線整理 |
| 軽さ・閉域適性 | 単一 EXE・外部依存なし・外部通信なし。起動即 TempNest。JSON 正本でユーザーがファイルを完全に掌握できる |
| 4 つの思考形式の並行利用 | ファイル単位タブ・ピン留め・detached window・タブ復元。Workspace 独立性が保たれ、1 つの操作ミスが他に波及しない |
| 操作への丁寧な応答 | ステータスバーの完了通知、ID-15 の新規カード位置フィードバック、CH-11/CH-18 の会話一覧の読みやすさ、ID-14 のフィルタ件数 |
| 開発持続性 | 責務文書（shell-partials / view・viewmodel responsibilities）、docs-contract テスト、静的テスト基準（TD-73）、1 version = 1 purpose の運用が実際に守られている（v2.18.0〜v2.18.12 の 13 リリースで確認） |

**「NestSuite の現在の最大の魅力は何か」（問い 1）への回答**: 複数の思考形式を、データ喪失の不安なしに、軽く・閉域で・単一 EXE で扱えること。特に「安心して雑に書き始められる」こと（TempNest 常駐 + 無題 draft 保護）は、書くことへの心理的障壁を実際に下げており、体験としての差別化点になっている。

## 3. 魅力を弱めている点

**「NestSuite の魅力を最も弱めている点は何か」（問い 2）への回答**: 書いたものが**書きっぱなしで終わる**構造。具体的には次の 4 点に分解できる。

1. **再訪の入口がない。** 起動時に見えるのは前回のタブと空の TempNest であり、「昨日何を考えていたか」「どのファイルが放置されているか」を示すものがない。SH-37 の状態サマリーは件数のみ・ヘルプメニュー配下であり、再訪の動機にはならない（現在地確認としては適切）。
2. **閉じたファイルは実質見えない。** 横断検索は開いているタブのみが対象（`CrossSearchPanel` の設計コメントで明示）。再発見の手段は「最近使ったファイル 10 件」と OS のファイラーだけで、NestSuite に蓄積するほど NestSuite の外（ファイラー・grep）が必要になるという逆転がある。
3. **成果物への出口が Workspace ごとに不均質。** ChatNest は会話コピー・Markdown・ファイル保存まで揃うが、IdeaNest のカード一覧出力は未実装（ID-10）、NoteNest・TempNest も粒度が異なる。「NestSuite で考えた内容を外に出す」体験が Workspace ごとに異なる操作になっている。
4. **利用者向け文書の鮮度が実装に追いついていない。** README がスキーマ `1.4.1` と記載（現行 `1.4.2`）、`.nestsuite` 標準拡張子（v2.14.1 FM-1）が README の形式表に未反映、ユーザーガイドは「引数なし起動は無題 NoteNest タブから始まる」（現行は TempNest アクティブ）、既知制約文書はヘッダーが v2.10.0 のまま「タブ並び替え未対応」「関連付け自動設定未対応」等の解消済み制約を残している。加えて `TutorialWindow`（`Assets/tutorial.png`）は `DialogService.ShowTutorial()` だけが残り呼び出し元がない。初見の利用者・評価者が最初に読むものが古いことは、実装の丁寧さを無効化する。

これらは個別機能の不足ではなく、方向性文書の 4 方向（再発見・流れ・ホーム・成果物）のうち **4.1（再発見）と 4.4（成果物）が未着手**であることの現れである。4.2（流れ）は TN-3、4.3（ホーム）は SH-37、4.5（愛着）は L23/ID-15/CH-11/CH-18/CH-16 が最初の一歩を済ませている。

## 4. 初回利用体験

起動フロー（実装から推定）: 起動 → TempNest タブがアクティブ → 空の 2×2 スロット（「タイトル」「一時メモ」のプレースホルダーと昇格ボタンの ToolTip のみ）。

| 評価項目 | 評価 | 根拠・補足 |
|----------|------|-----------|
| 何のアプリか直感的に分かるか | **弱い** | 起動直後の画面に「NestSuite が何か」を伝える要素がない。空の TempNest は「メモ帳の 4 分割」に見える |
| 4 Workspace の違いが分かるか | 中 | 新規作成メニュー・＋ボタンのメニューに一行説明があり（v2.15.1）、開けば分かる。ただし「開く前」に違いは伝わらない |
| どこから始めればよいか | 中 | TempNest が最初に開くのは「まず雑に書け」という暗黙の導きとして妥当。ただし意図として伝わってはいない |
| 空画面が不安・未完成に見えないか | 良 | L23（NoteNest）・IdeaNest・ChatNest の空状態ガイドは適切。TempNest はプレースホルダーのみで、Shell レベルの空状態（タブが Temp だけ）に案内がない |
| 最初の 1 件までの迷い | 良 | ＋ボタン・メニュー・空状態ガイドで到達できる |
| 初回 5 分で価値を感じられるか | 中 | 書く体験はすぐ成立する。「NestSuite ならでは」（4 形式の使い分け・保護の厚さ）は 5 分では伝わらない |
| 文書の第一印象 | **弱い** | §3-4 の鮮度問題。特に README は GitHub で最初に見えるページであり影響が大きい |

区別（説明不足 / 操作不足 / 視覚的魅力不足 / 役割理解不足）: 初回体験の課題は**役割理解不足**と**説明の鮮度不足**であり、操作不足・視覚的魅力不足ではない。したがってチュートリアルや説明文の追加ではなく、(1) 文書の鮮度修正、(2) Shell レベルの最小限の空状態一行（例: タブが Temp のみの初回起動時に、既存の新規作成導線を指す一行）で足りる。常設のウェルカム画面・大きなオンボーディングは認知負荷方針と衝突するため推奨しない。

**「初回利用体験に不足があるか」（問い 7）への回答**: ある。ただし不足は機能ではなく「役割の伝達」と「文書の鮮度」であり、最小コストで改善できる。

## 5. 日常利用・継続利用

| 評価項目 | 評価 | 根拠・補足 |
|----------|------|-----------|
| 前回の続きをすぐ再開できるか | 良 | session 復元・前回アクティブタブ復元・TempNest 常駐・無題 draft 復元 |
| 毎日開く理由があるか | **弱い** | 「開くと次に何を考えればよいか分かる」体験がない。開く理由は利用者の意思だけに依存している |
| 蓄積した情報を再発見できるか | **弱い** | §3-2。開いているタブの横断検索・NoteNest 内全ノート検索（M17 強調付き）は良いが、閉じたファイルに届かない |
| 再利用できるか | 中 | ChatNest のコピー/エクスポートは充実。NoteNest→ChatNest 引用・IdeaNest 出力等の出口は不均質 |
| 情報が増えるほど価値が上がるか | **弱い** | 増えるほど「どこに書いたか」が NestSuite の外の問題になる |
| 情報が増えるほど苦しくならないか | 良 | ファイル単位タブ・ピン留め・M14 並び順・IdeaNest フィルタ/アーカイブ・CH-11 日付区切りが密度増加に対応済み |
| 「書く→探す→見返す→整理する」の流れ | 中 | 「書く」は非常に強い。「探す」は開いているものに限り強い。「見返す」「整理する」の動機付けが弱い |
| 操作結果の気持ちよさ | 良 | ステータス通知・位置フィードバック・空状態の応答。§7 の Undo 不在のみが例外 |
| 保存・復元への不安 | 良 | §2 のとおり。過剰に表へ出てもいない（通知は失敗時のみ） |

**「毎日開く理由があるか」（問い 8）への回答**: 現状は「作業ファイルが NestSuite にあるから」以上の理由がない。TempNest 常駐と session 復元は「開いたときの摩擦ゼロ」を実現済みなので、次に必要なのは「開いた瞬間に続き・残り・放置が見える」入口（§10 AT-1）である。

**「蓄積した情報を再利用できるか」（問い 9）への回答**: 部分的にできる（ChatNest 出力・NoteNest リンク/検索）。閉じたファイルの再発見（§10 AT-2）と出口の均質化（§10 AT-3）が埋まれば、方向性文書の中心命題が一巡する。

## 6. Workspace 別評価

### NoteNest — 正式な記録の場: 役割明確・完成度高

- 正式なノート・記録としての役割は明確。ツリー・タスク（縮退方針で軽量維持）・マーカー・リンク・全ノート検索（M17 強調）・並び順（M14）と、長期保存・再読に必要な道具が揃っている。
- タスク縮退（TD-52 / RJ-10）後の右ペインは責務が明確になり、方針として正しかったと評価する。
- 残る改善は L4（ワードラップ切替）・L10（右ペイン絞り込み）等の実用品質であり、魅力の主戦場ではない。
- IdeaNest との差（構造化された長文 vs 断片カード）は実装上明確。初見への伝達は §4 の問題。

### IdeaNest — 断片を貯める場: 楽しさの余地が最も大きい

- カード・タグ・色・ピン留め・アーカイブ・フィルタ件数（ID-14）と、貯める・絞る道具は揃っている。過剰でもない。
- **「整理する楽しさ」に効く未実装が集中している**: Undo がないため削除・アーカイブのたびに確認ダイアログを挟む安全設計になっており、「気軽に触って整理する」体験を削っている（ID-6）。キーボードでカード間を移動できない（ID-4）。検索一致箇所が光らない（ID-7）。カード一覧を外に出せない（ID-10）。
- 「カードを次の作業へつなげる」導線は未着手（LK-3/LK-4 はトリガー待ち）。ただし §7 のとおり急がない。

### ChatNest — 一人ブレストの場: v2.18.10〜12 で完成度が上がった

- 発言者 4 種の切替（Ctrl+←/→）は「自分の考えに反論させる」思考補助として独自性があり、NestSuite の中で最も「らしい」Workspace。
- 日付区切り（CH-11）・視線誘導（CH-18）・操作整理（CH-16)により、長い会話の読み返しと入力集中の両方が改善済み。管理機能の増やしすぎもない（凍結方針が効いている）。
- 会話が長くなった後の価値remains: エクスポート・整形コピーが揃っており、4 Workspace で最も「出口」が強い。他 Workspace はこれを基準に揃えるべき（AT-3）。
- 残りの CH-17（送信前プレビュー）は、インライン編集・削除が既にあるため誤送信の回復コストが低く、**費用対効果が弱い**。優先度 B→C への引き下げを提案する（§9）。

### TempNest — 一時メモ: 役割は完成している

- 「閉じられない・保存対象外・起動時アクティブ」という設計は迷わず使える。TN-3 昇格で「放置されるだけ」への答えも最小実装済み。
- 昇格先の拡張（TN-7/LK-3）は TN-3 の実利用を見てからでよい（方向性文書どおり）。連携を増やすより、現状の「何も考えず書ける」軽さの維持が価値。
- TN-4（保存間隔・パス設定）は設定項目を増やすだけで魅力に寄与しない。長期保留のままでよい。

**「4 つの Workspace は適切な構成か」（問い 3）への回答**: 適切。追加も統合も不要。
**「各 Workspace の役割は十分に明確か」（問い 4）への回答**: 実装上は明確。初見への伝達（§4）だけが弱い。

## 7. NestSuite としての一体感

現状の一体感を支えているもの: ファイル単位タブ・共通保存系（Ctrl+S / Ctrl+Shift+S / `.bak` / draft）・共通テーマ・各 Workspace 共通の Ctrl+F・空状態ガイドの共通文化・ステータス通知・`.nestsuite` 統一拡張子（FM-1 wrapper）・「Nest」の命名体系。**単なる同居には見えない水準に既に達している。**

一方で不統一が残る箇所（利用者が気づき得るもの）:

- 出口の不統一: §3-3（ChatNest だけエクスポートが充実）。
- 検索の不統一は許容範囲（各 Workspace の Ctrl+F は対象が異なって当然）だが、横断検索が「開いているタブのみ」であることは説明されないと期待とずれる。
- `.nestsuite` 統一後も文書・UI 文言に旧 3 拡張子中心の説明が残る（§3-4）。

**「横断検索以外の一体感は必要か」（問い 5）への回答**: 必要。ただしそれは連携機能ではなく、(1) 再発見の共通入口（スタートパネル）、(2) 出口の共通言語（どの Workspace でも同じ場所に「Markdown でコピー/保存」）、(3) 文書・用語の一貫性で実現するのが NestSuite らしい。**疎結合を維持したまま一体感を高める方法は「データを繋ぐ」ではなく「入口と出口を揃える」**である。

**「Workspace 間連携を始めるべきか」（問い 6）への回答**: まだ広げない。TN-3 の実利用フィードバック待ちを維持する（方向性文書 §5・review7 §3.1 の判断を支持）。次に 1 本足すなら、思考の流れ（断片→発想→対話→記録）上で最も自然な **LK-4（ChatNest 発言→IdeaNest カード化）** を候補とするが、着手条件は「TN-3 が実際に使われている兆候」または「転送共通ヘルパー（review7 §4.1）の整備」とする。連携の数を増やすことは一体感に寄与しない。

## 8. 安心感・認知負荷

### 安心感（利用者）

§2 のとおり多層防御が完成しており、**これ以上の安全機能追加は不要**（追加はむしろ不安の演出になる）。唯一の隙間は「操作を試しても壊れない」方向の **Undo 不在**（IdeaNest の削除・アーカイブ、ID-6）。削除確認ダイアログは「壊れない」を保証するが「気軽に触れる」を犠牲にしており、Undo は確認よりも認知負荷が低い安全である。

### 認知負荷（利用者）

- 操作の入口過多は CH-16 で ChatNest について整理済み。他 Workspace に同水準の問題は見当たらない（IdeaNest の詳細画面は即編集方式に整理済み、NoteNest はメニュー構成が安定）。
- 設定は最小（テーマ・フォント・並び順）で健全。TN-4 のような設定追加は方針どおり抑制すべき。
- メニューの肥大化なし。ヘルプ配下（ショートカット・バックアップガイド・状態サマリー・関連付け・About）は適切な退避先として機能している。

### 認知負荷（開発者）

- Shell partial 分割（20 超のファイル）は `nestsuite-shell-partials.md` で追跡可能になっており、v2.18 系の 13 リリースが事故なく通っていることが構造の健全性の実証になっている。
- 静的テスト（docs-contract・XAML 契約）は TD-73/TD-75/TD-76 の集約で線形増加が管理されている。ただし **XAML 契約テストは version ごとに 1 クラスずつ増えており**（CH11/CH16/CH18/M14/L24…）、テストクラス命名方針（課題番号ベース回避）と緊張がある。次に同種テストを追加する際、Workspace 単位（例: `ChatNestWorkspaceViewXamlTests`）への統合を検討する価値がある（今回は変更しない）。
- release-notes（5,800 行超）は正本として機能しているが、docs-inventory 方針（TD-83）での定期的な棚卸しを継続すべき。

### v2.18.17 / AT-2: 横断検索の最近ファイル対象化

- **ID**: 新規（SH- 系採番）
- **目的**: 閉じたファイル（最近履歴 10 件）を明示スイッチで横断検索対象に含める
- **魅力への効果**: 蓄積の再発見。「書いておけば見つかる」への転換
- **実装範囲**: 検索パネルのスイッチ（既定 OFF）・未オープンファイルの読み取り専用検索・結果からの open
- **やらないこと**: インデックス作成（LT-2 不変）・全フォルダスキャン・履歴件数の拡大
- **設計レビュー**: **必要**（タブを開かない読み取り経路の切り出し。TD-59 で確立した「読込は 1 回」の契約を壊さない設計が必須）
- **実装後確認**: open 経路の回帰（二重読込ゼロの維持）・大きいファイル/破損ファイルのスキップ動作・検索性能

**「次の 3〜5 リリースは何を行うべきか」（問い 12）への回答**: 上記 5 件。低リスクの信頼回復（.13）→ 触る楽しさ（.14）→ 入口と出口の小物（.15）→ 再訪の入口（.16）→ 再発見（.17）の順で、失敗コストの小さい順に積み上げる。

## 15. 結論

- NestSuite は「壊れない・迷わない・軽い」を実装で証明済みであり、基盤の完成度は高い。**構成の再編・大規模設計変更・連携の一斉追加は不要である。**
- 最大の課題は「書きっぱなしで終わる」構造であり、**再訪の入口（AT-1）と再発見（AT-2）**がその答えになる。どちらも既存部品・既存情報・明示操作で実現でき、保存形式・外部依存・閉域方針に触れない。
- その前に、**低コストで確実な 3 件（文書鮮度 AT-4・IdeaNest Undo ID-6・出口の均質化 AT-3/ID-10）**を先に積み、失敗コストの小さい順に進める。
- CH-17・ID-13・TN-4 は見送り・保留へ倒し、schema 影響項目と転送拡張はトリガー待ちを維持する。
- 本レビューの施策が一巡した時点（または schema 影響・転送拡張の判断時点）で次回エキスパートレビューを行い、それまでの「すぐ実装」群は通常 backlog 運用で消化してよい。

以上。本レビューではソースコード・XAML・テスト・version・backlog・release notes・保存形式・schema・session/draft 形式・GitHub Actions・依存パッケージを一切変更していない（成果物は本文書 1 件のみ）。
