## TD-69〜TD-71 の評価

**TD-69（FilePaths を Tabs から導出）— 問題なし。**
- 変更前後で出力が同一であることの確認方法が適切。開いているタブ側は `TryCreateSessionEntry` / `TryCreateSessionTabState` がともに同じ `IsSessionPersistable` を通していたため、Tabs[] 起点への一本化で挙動差は生じない。pending entry の重複排除も、旧コードが `filePaths`（その時点で開きタブのみ）を渡していたのと新コードの `openTabFilePaths` は同値。
- TD-65 持ち越し: pending entry は Tabs[] へ統合後に FilePaths[] へ射影されるため、順序・内容・`WorkspaceKind = null`・`IsPinned` すべて維持。既存の重複排除テスト群（大文字小文字無視含む）が変更後も同一入出力になることも検証済み。
- 旧 FilePaths[] 互換: `CreateRestoreTargets` は無変更（Tabs 優先・FilePaths フォールバック）。TD-68 の「WorkspaceKind は書くだけ・読まない」も無変更。
- 副次的な利点として、**FilePaths[] と Tabs[] の長さ・順序の一致が構造的に保証**された。LT-9 で session を一覧表示する際、どちらを読んでも齟齬が出ない。

**TD-70（FileNotFound の再試行解除）— 妥当。設計判断 3 点すべて支持できる。**
- **利用者確認方式の採用は正しい。** FileNotFound はネットワーク/外部ドライブ未接続と恒久削除を区別できないため、N 回自動除外は「復元すべきものを勝手に落とす」リスクを持つ。確認ダイアログの文言も「移動・削除したなら はい / 一時的な未接続なら いいえ」と、誤って外しすぎない方向に誘導できている。
- **TD-65 の保護方針は壊れていない。** 解除は FileNotFound のみ・明示確認時のみ。InvalidFormat / SchemaVersionTooNew / AccessDenied は引き続き無条件で持ち越される。除外ロジックが UI 非依存の `SessionTabMapper.RemoveFileNotFoundEntries`（純粋関数・順序維持）に置かれ、挙動テストで直接検証できる形になっている点も良い。
- **`TryRestoreSession` が false でも保存する設計は正しく、むしろ必要だった。** session に FileNotFound entry しかない（復元 0 件）ケースで、解除の決定が保存されず次回また同じダイアログが出る、という穴をこの条件拡張が塞いでいる。フラグ 1 本 + 既存 SaveSession の条件拡張のみで、TD-66 の復元中保存抑止（`_isRestoringSession`）とも衝突しない（解除処理自体は SaveSession を直接呼ばないことをテストで固定済み）。

**TD-71（.bak 誘導 + 鮮度非対称の記録）— 妥当。役割混同なし。**
- .bak 誘導は「エラー通知メッセージの末尾 1 行・InvalidFormat 条件」、TD-70 確認は「別ダイアログ・FileNotFound 条件」、SH-1 は「起動時/外部オープンのみ・Open ダイアログと復元通知には不使用」— 3 者が異なる条件・異なる場所で発火し、それぞれ独立した静的テストで固定されている。ヒントは 1 メッセージ 1 回のみで、混在ケース（InvalidFormat + FileNotFound）でも「通知末尾に .bak 誘導 → 続けて解除確認」と順序が自然。
- ActiveFilePath の鮮度非対称を design-decisions §53 に「意図的な許容仕様」として理由（操作頻度・影響の小ささの比較）つきで記録したのは、review2 の趣旨どおり。docs-contract テストで記録の存在も固定されている。

## 新たなリスク

いずれも小さく、即対応不要です。

1. **【小】FileNotFound 継続ユーザーの起動時 2 ダイアログ化**。ネットワークドライブを毎回あとから接続する使い方の利用者は、「いいえ」を選び続ける限り毎起動で復元失敗通知 + 解除確認の 2 枚を閉じることになる。TD-70 導入前は 1 枚。将来の小改善候補として「復元失敗通知に解除選択を統合して 1 ダイアログにする」を backlog 化する価値はある（今は不要）。
2. **【小・記録推奨】InvalidFormat / AccessDenied / SchemaVersionTooNew には依然解除手段がない**。恒久的に壊れたままのファイルは毎起動で通知が出続ける。ただし間接的な逃げ道が実は存在する — ファイルを削除すれば次回から FileNotFound になり、TD-70 で解除可能。また TD-71 の .bak 誘導が復旧経路を示すため、「直すか、消して解除するか」の 2 択が案内上成立している。当面は許容でよいが、この間接経路をどこか（FAQ か将来の解除対象拡張の backlog メモ）に一文残すと親切。
3. **【小・仕様どおり】解除は all-or-nothing**。複数の FileNotFound entry を個別に選んで解除することはできない（一括のみ）。task 仕様どおりであり、個別選択は LT-9 の選択的復元 UI の領分。LT-9 設計時の要件として引き継ぐこと。
4. **【記録】静的テストの CI 破損が 3 回**。修正はいずれも根本的（CRLF 非依存境界・範囲パーサ・`— TD-NN:` 見出しアンカー）で、同型の再発は防がれている。ただし「後のバージョンの本文が前の ID を参照する」「範囲表記が圧縮される」といった、docs が自然に成長するだけで壊れるテストをこれ以上増やさないためのガイドライン化が望ましい（後述のエキスパート依頼参照）。
