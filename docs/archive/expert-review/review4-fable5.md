# LT-9 選択的 session 復元 UI 設計レビュー

## 総評

**LT-9 の前提条件は確かに揃いましたが、「揃った」ことと「今作るべき」ことは別です。** review1〜3 の対応（TD-65〜TD-71）によって、LT-9 が本来解決するはずだった痛点の大半 — 黙示的喪失・文言と実挙動の不一致・解除不能な nag — は既に別の手段で解消されています。残っている本物の需要は「**失敗 entry の個別扱い**」（TD-70 の all-or-nothing 制限と InvalidFormat 等の解除不能）に絞られており、これは「成功タブを選んで復元する」フル UI とは規模が一桁違う小さな問題です。

したがって結論は、**LT-9 フル UI は docs 設計メモに留め、その第一段階として SH-34（1 ダイアログ統合）を通常実装に切り出す**、です。LT-9 を「段階定義」に分解し、需要が実測されるまでフェーズ 3（成功タブの選択復元）を実装しないことを backlog に明文化することを推奨します。

もう 1 点、レビュー中に確認した重要な構造的制約があります: **session.json は失敗理由を保存していません**。失敗理由（FileNotFound / InvalidFormat 等）は起動時の `CreateRestoreTargets` がファイルを実際に調べて初めて分かる情報です。つまり「復元前に一覧を見せて選ばせる」UI は、表示のために全ファイルの事前スキャン（存在確認 + `.nestsuite` の種別判定読込）を要求し、TD-59 の二度読み問題を三度読みに悪化させます。「復元を試みた後、失敗分だけ見せる」方式ならこのコストはゼロです。これが UI 方針 B を強く支持する技術的根拠です。

## LT-9 を今実装すべきか

**フル実装は否。現在の自動復元 + TD-70 で日常利用は十分成立しています。**

- 「今日は前回のタブを開きたくない」ケース: 稀であり、復元後にタブを閉じるコストは小さい（ピン留めで守りたいタブは守れる）
- 起動性能の懸念: LT-11 の記録どおり実測された性能課題はなく、選択復元を性能対策として先回りする根拠がない
- 残る本物の需要は失敗 entry の個別解除のみ → SH-34/SH-35 の領域であり、フル UI 不要

## 推奨する UI 方針

**B（復元失敗 pending entry がある場合だけ表示）を推奨。**

- **A（毎回起動時）は不採用**: 正常起動は現在ダイアログ 0 枚。毎回選択を挟むのは「引き算のデザイン」方針への直接的な違反であり、99% の起動で無意味な決定を強いる
- **C（異常終了検知時）は不要**: TD-66 の随時保存により、クラッシュ後でもタブ構成はほぼ最新。異常終了を特別扱いする理由が消滅している
- **D（メニュー/Shift 起動から手動）は現構造と矛盾**: TD-66 以降、session.json は「現在の状態」を常時反映するため、起動後に「前回終了時の session」を選び直す元データが存在しない。D を成立させるには session snapshot（世代保存）という新機構が必要で、形式変更なし方針に反する。この不成立理由自体を docs に記録すべき（後述）
- **E は「フェーズ 3 についてのみ」採用**: 成功タブの選択復元は docs のみに留める

失敗時のみの UI は、現在すでにダイアログが出ている場面（復元失敗通知 + TD-70 確認の最大 2 枚）を 1 枚に**減らす**方向であり、認知負荷は増えるのではなく下がります。

## 表示対象と表示文言

フェーズ 1〜2 の表示対象は**失敗 entry のみ**。成功して開いたタブは表示しない（既に開いており、閉じれば済むため）。

| 項目 | 表示 |
|------|------|
| ファイル名 | `Path.GetFileName`（既存の複数件通知と同じ。フルパスは ToolTip 等の補助でよい） |
| 状態 | 「前回復元できなかったファイル」 |
| 理由 | `FileErrorMessages.ForKindDetectionFailure` の 1 行目（既存様式を再利用） |
| 種別 | `Tabs[].WorkspaceKind` ヒントを表示。null は「不明」 |
| Temp | 対象外（`IsSessionPersistable` により session に元々含まれない — 現行どおり） |

理由の情報源は「その起動での復元試行結果（`failures`）」であり、session.json に理由 field を足す必要はありません。

## pending entry / WorkspaceKind = null の扱い

- pending entry 由来の `WorkspaceKind = null` は「種別: 不明」と表示する。これは TD-65 が意図的に推測を避けた結果であり、UI 側で推測して埋めてはいけない
- 表示に `Tabs[].WorkspaceKind` を使うのは TD-68 が固定した「UI 表示ヒント」用途そのものであり、正当。ただし**復元ボタン押下後は従来どおり `CreateRestoreTargets` → `TryGetKind` の再判定を必ず通す**こと。UI で種別を表示したからといって再判定をスキップする最適化は、TD-68 のテストが守っている境界の侵犯になる
- InvalidFormat の entry には、TD-71 の `.bak` 誘導文（`MultipleFailuresBakDetailHint`）を同じダイアログ内に流用できる

## FileNotFound 解除・SH-34・SH-35 との関係

- **SH-34 は LT-9 のフェーズ 1 として統合すべき**。「復元失敗通知 + 解除確認の 1 ダイアログ化」は、選択的復元 UI の最小の種であり、独立タスクとして先行実装する価値がある（MessageBox の文言統合だけで済み、カスタム Window すら不要）
- **FileNotFound の個別解除はフェーズ 2**（小さなカスタムダイアログ + entry ごとのチェックボックス）。TD-70 の all-or-nothing 制限の解消はここで行う
- **InvalidFormat / AccessDenied / SchemaVersionTooNew の解除拡張（SH-35 の実装部分）もフェーズ 2 に吸収**するのが自然。同じダイアログに並ぶ以上、解除可否を理由種別で分ける複雑さより、「全種別解除可能・ただし InvalidFormat には強い注意文言（.bak 復元を先に案内）」の方が一貫する。ただしこの判断はフェーズ 2 設計時に再確認すること
- **SH-35 の FAQ / 間接経路記載部分は独立の docs タスクとして別管理のまま**でよい（実装を待つ必要がない）

## session.json 形式変更の要否

**不要（フェーズ 1・2 とも）。**

- 一覧表示: `Tabs[]`（FilePath / WorkspaceKind ヒント / IsPinned）+ 起動時の `failures` で完結
- 解除: 既存機構（保存時に pending から除外するだけ）の対象指定を「全 FileNotFound」から「選択された entry」へ一般化するのみ。`RemoveFileNotFoundEntries` をパス集合指定版へ拡張すれば足りる
- `retryCount` / `lastFailureTime`: 自動除外（既に TD-70 で棄却済みの方式）にしか使い道がなく、**将来も不要の見込み**
- `displayName`: `Path.GetFileName` で足りるため不要
- 唯一形式変更があり得るのはフェーズ 3（成功タブの選択復元で「前回選択状態の記憶」等を持つ場合）だが、そこまで進む根拠が現状ない

## ActiveFilePath / LT-4 との関係

- **ActiveFilePath**: 初期実装では現行どおり「復元成功タブのうち前回アクティブだったものを選択」で足りる。選択 UI で「最初に開くタブ」を選ばせる必要はない。§53 の鮮度非対称はフェーズ 1・2 に一切影響しない（失敗 entry はそもそもアクティブになれない）
- **LT-4**: フェーズ 1・2 は `Tabs[]` 単一 Window 前提で問題ない。将来 `Windows[]` が追加された場合、失敗 entry 一覧は Window 非依存（ファイル単位）のため UI 変更不要。フル選択 UI（フェーズ 3）を作る場合のみ Window 別グルーピングが論点になるが、リスト UI はヘッダ追加で自然に拡張できる。**今の段階で Windows[] を前提にした UI 設計はしないこと**（§54 の方針メモどおり、境界だけ守れば十分）

## 認知負荷評価

- 正常起動: ダイアログ 0 枚 → 0 枚（不変）。ここを崩さないことが最重要
- 失敗あり起動: 現在最大 2 枚 → フェーズ 1 で 1 枚。**負荷は純減**
- 個別チェックボックス（フェーズ 2）は「失敗がある人だけ」が見る画面なので、全体の認知負荷への影響は限定的。ただし既定値は「すべて未チェック（= 従来どおり再試行）」とし、安全側に倒すこと
- 「前回 session の復元をやめる / 続ける」の二択 UI: 一見シンプルだが毎回起動時に出すなら A 案と同じ問題を持つ。失敗時のみなら「やめる」の意味が曖昧（成功分は既に開いている）。採用しない

## 最小実装案

**フェーズ 1（= SH-34、通常エンジニア可、即着手可）:**
`NotifyRestoreFailures` と `OfferToForgetFileNotFoundRestoreFailures` を統合。FileNotFound を含む場合は、失敗一覧 + TD-71 の .bak 誘導 + 解除確認文を 1 つの Yes/No MessageBox にまとめる。FileNotFound を含まない場合は従来どおり OK のみの通知。コードは文言結合と分岐整理のみ、形式変更なし、`_forgotFileNotFoundRestoreFailuresDuringStartup` の保存経路もそのまま。

**フェーズ 2（LT-9 最小形、フェーズ 1 の実績を見てから）:**
失敗 entry 専用の小さなカスタムダイアログ（ListBox + チェックボックス）。列: ファイル名 / 理由 / 種別（null は「不明」）。チェックした entry を「次回から再試行しない」。全失敗種別を解除可能にし、InvalidFormat には .bak 案内を添える。解除は `SessionTabMapper` の一般化 helper（選択パス集合を除外）で行い、UI 非依存の挙動テストを付ける。

**フェーズ 3（実装しない・docs のみ）:**
成功タブを含むフル選択復元。実測された需要（起動性能の苦情、または「毎回タブを閉じ直している」というフィードバック）が出るまで凍結。

## 実装しない方がよい案

- 毎回起動時の選択 UI（A）— 方針違反かつ 99% の起動で無価値
- 復元前の事前スキャン方式 — 失敗理由の表示のために全ファイル読込が必要になり、TD-59 を悪化させる。「試行後に失敗分だけ」方式が構造的に優位
- session snapshot / 世代保存による「後から選んで復元」（D）— 新永続機構が必要で、TD-66 の「session = 現在の状態」という意味論と衝突する
- retryCount 等による自動除外の復活 — TD-70 で棄却済み。明示確認方式を維持
- WorkspaceKind を表示以上に使う最適化 — TD-68 の境界侵犯

## 特に知りたい結論への回答

1. **LT-9 フル実装はまだ docs 設計メモに留める。** ただし第一段階（SH-34）は通常実装として次に着手してよい
2. 実装するなら**失敗時のみ UI（B）**。毎回起動時（A）は方針違反、手動呼び出し（D）は session snapshot が必要になり現構造と矛盾するため不採用

4. **session.json 形式変更は不要**（フェーズ 1・2 とも既存の Tabs[] / FilePaths[] / ActiveFilePath で成立。失敗理由は起動時再判定で得られるため新 field 不要）
5. **SH-34 は LT-9 のフェーズ 1 として統合。SH-35 は分割** — FAQ / docs 部分は独立タスクとして別管理のまま、解除対象拡張部分はフェーズ 2 で吸収する
